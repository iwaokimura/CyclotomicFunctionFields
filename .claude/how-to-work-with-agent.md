# コーディングエージェントとの上手な働き方

2026-03-08 の blueprint CI 作業の振り返りから。

---

## 1. 外部ツールはバージョンを固定させる

`@main` や `@latest` を使ったアクション・パッケージは、自分たちのコードを何も変えていないのに突然壊れることがある。

**指示の仕方**:
> 「このアクションを SHA かタグにピン留めして」

エージェントはデフォルトでは最新を使いがちなので、明示的に指示する。

---

## 2. 試行錯誤は feature branch でやらせる

CI を使った試行錯誤（特に設定ファイルの調整）を `master` に直接コミットさせると、履歴が汚れる。最初から伝える。

**指示の仕方**:
> 「feature branch を切って、そこで作業して。CI が通ったらマージして。」

今回は後半でこの指示をしたが、最初から言えばよかった。

---

## 3. エラーログはエージェントに読ませる前に自分でも確認する

エージェントはログの一部だけ見て原因を誤診することがある。今回は `Replaying ... style.css not found` というログが「クラウドキャッシュからのリプレイ失敗」を意味していたが、「docgen-action のバグ」と誤判断した。

**習慣**:
- `gh run view <id> --log` の出力を自分でも眺めて、気になる単語（`Replay`, `cache`, `Permission denied` など）があれば指摘する。
- エージェントが「原因はXです」と断言しても、「本当にそうか？」と一度問い返すとよい。

---

## 4. まず「成功している他のプロジェクト」の設定を探させる

外部ツールの使い方に詰まったとき、ドキュメントやソースを読む前に、**同じツールを使っている他のプロジェクトの設定ファイルを探す**ほうが解決までの道が短い。

**指示の仕方**:
> 「このツールを使って同じことをやっている Mathlib 系プロジェクトを探して、そのワークフローを参考にして」

今回の例：`docgen-action` で blueprint + API docs を両立している他プロジェクトのワークフローを最初に調べていれば、`build-page:false` という解法にすぐたどり着けた可能性が高い。ソースコードを読んで解決したが、それは遠回りだった。

なお「有名な参照例を探す」→「見つからなければソースを読む」という順序が効率的。

---

## 5. 「最小の変更」を意識させる

動いていた設定を捨てて別のアーキテクチャに切り替えるのは、未知の問題を新たに生む。

**指示の仕方**:
> 「最小の変更で直して。動いていた部分はなるべく触らないで。」

エージェントは「きれいな解決策」を好む傾向があり、動いていたものを捨てて全部書き直しがちなので、意識的に抑える。

---

## 6. CI に頼る前にローカル検証できないか確認する

CI は 1 run に 15〜30 分かかる。ローカルで確認できることはローカルでやった方が何倍も速い。

**指示の仕方**:
> 「CI に投げる前に、ローカルで確認できることはある？」

例：
- `leanblueprint web` → `blueprint/web/` が生成されるか
- `docs/` の構造が正しいか
- `lake exe checkdecls` が通るか

---

## 7. 作業前にメモを書かせる / 定期的に状況を整理させる

長い作業になると、エージェントも（会話履歴が長くなるにつれ）前の判断を忘れたり、同じ失敗を繰り返すことがある。

**有効な指示**:
> 「今日やったことをメモにまとめて」（`.claude/memo.md` など）
> 「次に何をするか計画を立てて」
> 「この問題に対してどんなアプローチがあるか、やる前に挙げて」

特に詰まったときは、「一度整理して」と言うと有効。

---

## 8. issue を立ててから作業させる

何かを修正・実装するとき、先に issue を立てさせると：
- 問題の理解が整理される
- 作業範囲が明確になる
- 後から「何のための変更か」が追跡できる

**指示の仕方**:
> 「まず issue を立てて、それから branch を切って作業して」

---

## 9. 大きな決断はエージェントにやらせる前に確認する

「`docgen-action` を完全に削除する」のような、後から取り戻しにくい変更は、エージェントが自己判断でやるのではなく、**計画を先に提示させて同意してから実行**させる。

**習慣**:
- 「計画を立てて」→ 確認 → 「実行して」の 2 ステップを踏む
- 特にファイルの削除、依存の削除、アーキテクチャの変更は要注意

---

## まとめ

エージェントは**実装は速い**が、**判断は人間が監督する**という役割分担が基本。
特に「外部環境の変化への対応」や「トレードオフの判断」は人間が主導する方がうまくいく。
